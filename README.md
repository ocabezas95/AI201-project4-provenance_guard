
# Provenance Guard — Production Core API

An automated multi-signal ingestion backend pipeline that evaluates the provenance of text submissions, manages human-readable uncertainty representation bands, and handles creator status appeals safely at volume.

---

## 🏗️ Architectural Overview
A text payload submitted to `POST /submit` flows through the system using the following linear pipeline:
1. **Intake & Validation Guard:** Filters incoming request shapes and blocks submissions falling under the `< 40` words minimum length limit to protect upstream API costs.
2. **Multi-Signal Execution Engine:** Fires parallel requests to **Signal 1 (Semantic)** via Groq (`llama-3.3-70b-versatile`) and **Signal 2 (Structural)** via native Python stylometrics.
3. **Uncertainty Evaluation Layer:** Combines signal responses using a $60/40$ weighted average. If a severe signal conflict arises ($|S_1 - S_2| > 0.5$), the system activates a **Disagreement Override** to push the item into a protected intermediate category.
4. **Persistence & Log Write:** Records metrics atomically within an indexed SQLite instance (`provenance.db`) before rendering the final JSON contract response.

---

## 📡 Active Detection Signals
* **Signal 1 (Semantic Pacing):** Leverages Groq's `llama-3.3-70b-versatile` model at `temperature=0` to evaluate thematic cohesion, rhythmic phrasing, and generic sentence burstiness signatures.
    * *Linguistic Blind Spot:* Highly polished academic writing, legal summaries, or formal standardized texts that mimic structural predictability.
* **Signal 2 (Native Stylometric Heuristics):** Compares local statistical parameters using native Python libraries without external engine overhead. It runs a sliding 50-token **Moving Average Type-Token Ratio (MATTR)** to evaluate lexical diversity, and measures the **Sentence Length Coefficient of Variation (CV)** to detect robotic sentence distribution patterns.
    * *Linguistic Blind Spot:* Intentionally repetitive human expressions, such as stylized poetry or structured matching list formats.

---

## 📊 Confidence Calibration & Real Score Verification
The system utilizes explicit intervals to map combined final scores into non-jargon, human-readable bands:
* **Score < 0.40:** Likely Human-Written
* **0.40 ≤ Score ≤ 0.70:** Uncertain / Mixed Signals
* **Score > 0.70:** Likely AI-Generated

## 🛡️ Production Safety Configuration & Limits

### 1. Rate Limiting Strategy
* **Route Targets:** `@limiter.limit("10 per minute; 100 per day")` applied to `POST /submit`.
* **Engineering Rationale:** An ordinary human author posting original creative prose on a publishing portal will rarely submit text updates faster than once or twice a minute. Restricting the endpoint to 10 requests per minute leaves generous headroom for active workflows while blocking adversarial scripting loops from exhausting expensive upstream LLM tokens.

## 🤖 AI Collaboration Reflection
**1. (Prompt Alignment):** Instructed the AI assistant to scaffold the stylometric statistical functions. The model initially proposed a raw variance routine, but upon a joint review, we modified the code to utilize the scale-invariant Coefficient of Variation (CV). This choice guarantees structural fairness across both long and short text submissions.

**2. (State Machine Enforcement):** Directed the assistant to build out the reviewer /resolve endpoint. I overrode the model's initial unrestricted updates to introduce a strict state-transition boundary check `(row["status"] == "under_review")`, preventing resolved tickets from being modified after hitting a terminal state.
